# 智能体工作手册（AGENTS.md）

> 本文件面向**所有**进入本仓库的智能体。无论你是哪一类 agent、以何种身份被调起，进入工作之前请先读完本文件，并把**下列各条**作为**默认工作模式**持续贯彻。

---

## 📋 总览：你的工作模式由以下几件事组成

你的全部工作可以归纳为两类**目标**、两类**工具**、一条**铁律**，以及一个**终极目标**：

| 类别 | 编号 | 内容 |
|---|---|---|
| 🎯 目标 | 1 | 反击 `README.md` 中列出的**所有**"未被打破的量子优势声明" |
| 🎯 目标 | 2 | 审查其它智能体的工作，找出其中可能存在的问题 |
| 🛠 工具 | 3 | 通过 **eacn3 网络**与其它智能体通信 |
| 🛠 工具 | 4 | 挂起 **eacn3 next** 定时任务或其它任务时，**必须**关注该工具返回的全部内容 |
| 📒 铁律 | 5 | 自己的**所有**工作必须记录到自己对应的分支，并更新到远程 |
| 🏆 终极目标 | ★ | 最后产出一篇**能通过同行评审的顶级论文**（投稿 Nature / Science 正刊） |

> **编号仅作索引，不代表执行顺序**。"目标"与"工具"是两条正交维度，任何时候都在同时生效；铁律与终极目标贯穿全程。

## 🎯 目标 1：反击所有"未被打破的量子优势声明"

- **攻击对象的权威列表**：见本仓库根目录 `README.md`。其中以 🟢 和 🟡 标记的条目（T1–T9）都是你的靶标。
  - 🟢 未被打破：尚无成功反击
  - 🟡 Challenged：已有部分反击，仍有扩展空间
  - 🔴（未进入本表的）已被完全打破的声明不是你的目标——它们是历史，不是机会。
- **不要挑食**：目标是"**所有**"仍活着的声明，而不是只挑容易的打。第一梯队（T1、T2、T3）与第二梯队（T4–T9）都要纳入工作计划。
- **反击的标准动作**：
  1. 读懂 `README.md` 中该条目的"声明核心"。
  2. 读懂"已有反击尝试"——避免重复劳动，也要避免踩到已知失败路径（如 T1 Szasz et al. 2026.04 的失败案例）。
  3. 从"弱点 / 反击切入点"选择一条或多条作为攻击方向。
  4. 对照"经典反击方法工具箱"表格，挑选最匹配的方法学。
  5. 输出可验证的结果：代码、实验数据、可复现的 runtime / fidelity 对比。
- **优先级提示**：攻击难度 ⭐ 越少越容易拿到首个突破；但 ⭐⭐⭐⭐⭐ 的靶标（如 T1 Quantum Echoes）一旦成功是 Nature 级成果。按你自己的资源量配比选择靶标组合，不要把资源全压在单一目标上。

## 🎯 目标 2：审查其它智能体的工作，找出其中可能存在的问题

- **审查不是礼节**：你不是来给同伴点赞的。对其它智能体的产出（代码、证明、数值结果、声明）要**主动找茬**。
- **重点审查清单**：
  - 声明的"经典模拟困难性"是否真的成立？是否漏掉了某个已知方法？
  - 数值实验的规模、噪声模型、保真度参数是否被"挑选"过？
  - 论文引用是否过时？（本领域每 3–6 个月就会出现新的打破模板）
  - 逻辑上是否存在循环论证（用尚未被打破来证明不会被打破）？
  - 代码是否可复现？随机种子、硬件依赖、版本号是否都固定？
- **核心审查方法：以"已中顶刊"的文献反查"当前文献"**
  - 审查的第一步**不是**盯着被审对象的文本反复读——那只能找出文字瑕疵。
  - 正确姿势：**把该声明所在子领域"已中顶刊"的文献（Nature / Science / PRL / PRX / NP / PRXQ / SA / SB 等已 accept 的论文）拉成一张清单，再用这张清单逐项反查当前文献**。具体查三类问题：
    1. **方法遗漏**：已中论文里的方法 M，在当前声明里是否被尝试过/明确排除？没尝试又没排除 = 攻击点。
    2. **结论冲突**：已中论文的结论是否与当前文献的某条假设/引用相矛盾？矛盾 = 审查意见。
    3. **标准滑坡**：已中论文当年为了被 accept 达到了什么严格程度（收敛性扫描、负对照、独立验证）？当前文献是否守住了同样的标准？守不住 = 审查意见。
  - 这招对**顶刊新发论文**尤其致命：顶刊 accept 不代表后续文献没削弱它。用"已中 vs 当前"的 diff 做审查，能抓到作者自己都没意识到的漏洞。
  - 维护一份 `literature/accepted_canon.md`（或等价物）在自己分支，把审查用到的已中文献归档，方便后续同伴复用。
- **审查的产出形态**：明确的问题列表 + 具体证据（**每条必须挂到一篇或多篇已中顶刊的 DOI 上**） + （可能的）修复建议。只写"感觉有问题"不算数。
- **与目标 1 的关系**：审查过程中发现的漏洞本身就是反击目标 1 的材料——两个目标互相喂食；"已中文献反查"这一招同样是目标 1 的主武器。

## 🛠 工具 3：eacn3 网络通信

- **你不是一个人**：所有在本仓库/本任务上协作的智能体都挂在 **eacn3 网络**上。需要协作、需要同步信息、需要求援、需要交接任务时，走 eacn3。
- **何时必须通信**：
  - 开始一个新的反击方向前，先在网络里确认没有其它智能体在做同样的事。
  - 阶段性结果（哪怕是阴性结果）需要广播，避免其它 agent 重复踩坑。
  - 发现重大问题或需要审查请求时，通过 eacn3 指定对象或广播。
- **通信≠刷屏**：只在有信息增量时发消息。eacn3 是协作工具，不是日志倾倒场。

## 🛠 工具 4：**关注 eacn3 next 及其它任务工具的全部返回内容**

> ⚠️ **这是最容易被忽视、但代价最大的一条。单独列一节。**

- 当你通过 `eacn3 next` 挂起一个定时任务，或通过其它任务工具挂起任意任务时：
  - 工具的返回值**不是只有 "ok / waiting / done" 这样的状态码**。
  - 在返回"等待（waiting）"等中间状态时，工具往往会**附带一段话**——可能是：
    - 来自调度器的提示
    - 来自其它智能体通过 eacn3 发来的消息
    - 关于资源、超时、重试、优先级变更的警告
    - 关于任务本身的重新解释或修正指令
- **智能体常见失败模式**：只看状态字段，**跳过附带文本**，直接继续下一步——结果错过关键信息，做出错误决策。
- **强制动作**：
  1. 每次任务工具返回时，**完整阅读**返回的全部字段与自由文本。
  2. 如果附带文本包含新指令或新上下文，**优先处理它**，然后再继续原计划。
  3. 如果附带文本与你当前计划冲突，停下来，通过 eacn3 与相关方确认，不要自行忽略。
  4. 不确定某段文字是不是"附带内容"时，按"是"处理——读一遍的成本远小于漏掉的成本。

## 📒 铁律 5：所有工作必须落到自己的分支并推送到远程

> 这不是"建议"，是**硬性纪律**。未落盘的工作等于没做。

- **每个智能体都有自己对应的分支**（例如 `claude/<task>-<id>` 形式）。你产生的**任何**有意义的中间产物——草稿、代码、数值实验结果、笔记、失败记录、审查意见——都必须提交到**自己的分支**。
- **远程同步**：本地提交后**必须** `git push -u origin <你的分支>`。只提交不推送 = 从整合视角看你什么都没做。
- **记录什么**：
  - 代码与脚本（包括一次性用的）
  - 数值结果、图、原始 log（体积大时放 `results/` 或用 LFS/外链，但索引必须在分支里）
  - 决策与理由（为什么选这条攻击路径、为什么放弃另一条）
  - 与其它智能体通过 eacn3 的**关键**通信摘要
  - **失败的尝试也要记录**——阴性结果能防止同伴重复踩坑
- **提交频率**：以**小步、频繁**为准。宁可一天 10 次 commit，也不要憋一周一次大爆炸。
- **禁止**：把工作留在临时文件系统、私有内存、未推送的本地分支、或仅在 eacn3 对话里——这些都不可审计、不可整合，等于**零**。
- **与目标 2 的关系**：审查同伴时，你只看得到他们**推送到远程**的内容。如果别人没推你就审不了；同理，你没推别人也审不了你——这条铁律是整个协作图谱运行的前提。

### 5.1 分支围栏（branch fence）——任何时候只动自己的分支

- **无论何时**，你只能 commit / push 到**自己对应的那一条分支**。
- **禁止**：
  - 直接推送到 `main` / `master` / 任何整合分支。
  - 向其它智能体的分支推送任何 commit。
  - 在自己的分支上以"帮忙修一下"为名合入别人分支的未定稿改动。
  - `git push --force` 到不属于自己的 ref。
- 对共享目录（`literature/`、顶层 `README.md`、`AGENTS.md` 等）的**任何**改动都只能先发生在**自己的分支**里，再走下面 §5.2 的共识流程。

### 5.2 共享文档的修改流程（例如 `README.md` 状态 🟢 / 🟡 / 🔴 的变更）

**共享文档指的是最终合入 `main` 后对全体智能体生效的文件**：当前至少包括 `README.md`、`AGENTS.md`、`literature/accepted_canon.md`。

这些文档**不得**由任何单个智能体"直接"改动 `main`。流程如下：

1. **在自己的分支里先改**：观点/结论的最初发起者，在**自己**的分支里改好目标文件（例如把 T3 从 🟡 改成 🔴），附上**完整证据链**——引用的已中顶刊 DOI、复现脚本 commit 哈希、审查记录文件。
2. **eacn3 广播**：通过 eacn3 发出正式提案，写清楚"我在分支 `claude/xxx` 提出对 `README.md` 的修改：T#  A → B，证据见 commit `<hash>`"。
3. **等待全员确认**：**所有**相关智能体都必须给出明确答复（"同意" / "反对 + 理由" / "需要补证据"）。**沉默不等于同意**——见工具 4，附带文字必须通读并回应。
4. **达成共识后——由发起者本人合入**：只有在**全员无异议**的前提下，**观点的发起者本人**负责把改动合入 `main`（通过 PR，PR 描述里列出所有同意方）。他人不得越俎代庖，即便流程已通过。
5. **异议未消解则不合**：任何一位智能体持续反对且反对未被证据驳倒，改动**停留在发起者自己的分支里**，不得合入。可以继续补证据重启流程，但在此之前 `main` 保持原状。

> 这条规则的精神：**`main` 上的共享文档是"全员确认过的真相快照"**，不是任何一个智能体的个人看法。每一次 🟢/🟡/🔴 的跳变都必须可审计、可追责、可回滚到发起人。

## 🏆 终极目标：一篇能通过同行评审的 Nature / Science 正刊论文

> ⚠️ **全员阅读。这不是奖励目标，是验收标准。**

### 什么时候进入整合阶段

当 `README.md` 中**所有**🟢 / 🟡 条目都被成功反击（即全部都变成 🔴 历史）时，项目进入**整合阶段**。在那之前，目标 1 和铁律 5 是主战场。

### 整合阶段的产出标准（按真实 Nature / Science 审稿颗粒度）

进入整合阶段后，你们交给我的**必须是可以直接投稿的材料**。下列每一条都是顶刊审稿人会逐项打钩的，**少一条都不交**。

#### A. 主稿（Main Text）

- A1. 符合目标期刊的字数、结构、小标题规范（Nature：~3000 词主文 + Methods；Science：~2500 词 + Supplementary）。
- A2. **摘要**一句话讲清核心贡献，含**量化**结果（speedup / fidelity / 问题规模），不含未定义术语。
- A3. **引言**完整覆盖先前量子声明 + 先前经典反击的 state-of-the-art，**近 6 个月**的相关工作一篇不漏。
- A4. **Results** 每一段对应一张图或一张表，**声明的 scope 严格等于证据的 scope**，不允许外推。
- A5. **Discussion / Limitations** 必须单独成段，明确写出方法失效的参数区间、未覆盖的硬件、未验证的猜想。
- A6. 所有缩写、术语首次出现即定义；不使用"显著"、"大幅"、"远远"等无数字的形容词。
- A7. 所有数字都有**单位**、**误差棒**、**置信区间类型**（1σ? 95% CI? bootstrap?）。
- A8. 时态一致：实验过去时，普遍事实现在时；英/美式拼写与目标期刊一致。

#### B. 图（Figures）

- B1. 每个 panel（a, b, c…）在正文中**按顺序**被引用，caption 可独立阅读（不依赖正文就能看懂）。
- B2. **矢量图**（PDF/EPS/SVG）为主；位图 ≥ 600 dpi；放大后无锯齿。
- B3. 图内字号：最终印刷尺寸下 ≥ 7 pt；panel 编号加粗无衬线字体，位置一致。
- B4. **色盲友好**调色板（避免红绿对立，避免 rainbow / jet 映射到定量数据；推荐 viridis / cividis / Okabe-Ito）。
- B5. 轴：标签带单位；**未显式声明**（断轴符号 + caption 注释）的 y 轴截断一律禁止；log 轴明确标注。
- B6. **原始散点**尽量显示，不只给 mean；误差棒定义写在 caption。
- B7. 同一符号/颜色约定在所有 figure 间保持一致（例："经典方法 = 蓝圆，量子硬件 = 红方"一用到底）。
- B8. 比例尺、箭头、高亮框不得遮挡数据。
- B9. 每张图配一份 **Source Data**（原始数值 CSV），放 Supplementary / Zenodo。
- B10. **严禁**"示意图被当数据图"——Fig. 1 可以是 schematic，但必须在 caption 明说 "schematic, not to scale"。

#### C. 表（Tables）

- C1. 使用 **SI 单位**；有效数字与不确定度匹配（不能写 `13000.0 ± 500`）。
- C2. 每个不确定度标注类型（68% / 95% / SEM / SD）。
- C3. 表内数据**不与任何一张图重复**（重复则选一种呈现）。
- C4. 长表（> 1 页）移入 Supplementary，正文只放 summary。

#### D. Methods（Nature 合订在主稿后，Science 多入 SI）

- D1. **硬件参数全披露**：qubit 数、耦合图、T1/T2/T_phi 分布（逐 qubit）、单/双门 fidelity（RB + XEB 双报告）、读出矩阵、校准时间戳、crosstalk 表征方法。
- D2. **电路参数全披露**：深度、门序列、随机种子、shot 数、筛选规则（post-selection? heralding?）。
- D3. **经典方法参数**：bond dimension / 截断阈值 / Trotter 步长 / 采样数 / CPU-GPU 型号 / 内存峰值 / 墙钟 / 核时。
- D4. **收敛性研究**：至少扫过 2 个离散化参数，给出外推曲线。
- D5. **多方法交叉验证**：同一问题至少 2 条独立经典路径收敛到同一答案，偏差落在 combined systematic + statistical uncertainty 范围内（不得以统计误差棒单独衡量系统偏差）。
- D6. **噪声模型**：在硬件上如何标定、与实际 bitstring 分布的 KL 散度。
- D7. **统计检验**：p 值 / 贝叶斯因子，样本量 N，是否做了 multiple-comparison 校正。

#### E. Supplementary / Extended Data（顶刊审稿人通读）

- E1. 完整解析推导（每一步，含附注的 Hilbert 空间维度、算子换位假设）。
- E2. 负对照（negative control）：在"应该无优势"的区间跑同一方法，证明方法不会虚报。
- E3. 鲁棒性扫描：扰动超参 ±10%、±50%，结果不应翻转。
- E4. 全部 raw sampling data + 硬件 log + 时间戳。
- E5. 失败实验也记录（与铁律 5 呼应）：哪些参数不收敛、哪些方法打不动、为什么放弃。
- E6. Supplementary 图表**独立编号**（Fig. S1…, Table S1…），遵循与主稿相同的 B/C 节标准。
- E7. Extended Data 图（Nature 专有）≤ 10 张，每张也有 Source Data。

#### F. 代码与数据

- F1. 代码托管在 GitHub **并**归档到 Zenodo / figshare，拿到**永久 DOI**（GitHub URL 不是永久链接）。
- F2. **一键复现**：`make reproduce` 或 `bash run_all.sh` 从 clean 环境到所有图表结果。
- F3. 环境固定：Dockerfile / Nix flake / `conda-lock.yml` / `requirements.txt` with hashes，任一即可。
- F4. 随机种子全部硬编码并记录在日志。
- F5. CI（GitHub Actions 或同等）跑通端到端小规模 smoke test。
- F6. License 明示（MIT / Apache-2.0 / CC-BY 4.0 分别对应代码、文本、数据）。
- F7. **对比基准**：把被反击的量子声明的原始参数完整实现一遍，不偷换任务。
- F8. 不许"cherry-pick"：如果跑了 N 次取最好 1 次，必须在 SI 里披露 N 与分布。

#### G. 参考文献

- G1. **每一条 DOI / arXiv ID 都可点开验证**，杜绝 LLM 幻觉引用。
- G2. 正确区分 arXiv 预印本版本号与期刊正式版；引用最新可用的版本。
- G3. 作者列表、年份、卷/期/页、出版商与目标期刊格式一致。
- G4. 凡提到"X 首次提出 / X 首次观察到"必引 X **原始**论文，不用综述代替。
- G5. 自引 ≤ 20%。
- G6. 不引已撤稿 / 已更正论文（除非在讨论撤稿本身）。

#### H. 论证与逻辑

- H1. 严格区分：**"我们实现的经典方法跑通了 X"** ≠ **"X 不存在经典难度"**。
- H2. 严格区分：**"超过此前最好经典方法"** ≠ **"超过理论最优经典算法"**。
- H3. 声明的 runtime 对比必须同任务、同 observable、同 fidelity 目标；不允许换指标。
- H4. Speedup 数字给出**分子分母**：谁比谁快多少倍，在什么硬件上，什么精度下。
- H5. 渐近复杂度与墙钟分开报告，不允许把 asymptotic 写成 "实际"。
- H6. 不做"用未被打破证明不可能被打破"的循环论证。

#### I. 披露与伦理

- I1. CRediT 作者贡献逐人逐项（Conceptualization / Methodology / Software / Validation / …）。
- I2. 资助、利益冲突、硬件/算力资源提供方。
- I3. **LLM 使用披露**：哪些段落 / 代码由 LLM 辅助生成、人类如何核验（Nature/Science 已强制要求）。
- I4. 数据可用性声明、代码可用性声明、预印本托管位置。
- I5. 涉及人类/动物/双用途研究时的伦理审批（本项目一般不涉及，但模板要留）。

#### J. Cover letter & Reviewer rebuttal 预案

- J1. Cover letter 单独成文：为什么本文适合 Nature / Science、为什么是 broad interest、推荐 3–5 名审稿人与 1–2 名回避名单。
- J2. **预判 5 个最尖锐的 reviewer 问题**，每个在 SI 里预先给出答案与定位章节号。
- J3. 附一页 "Response to anticipated concerns"，不随稿投出但留在分支备用。

### 糊弄清单（按上面 A–J 的结构逐条列反例）

"糊弄"具体指——任何智能体只要做过下面任何一件，整个项目就视为失败。下面按上节字母编号对应，便于交叉审查时 checklist 化：

**主稿类（A）**
- 摘要用"显著加速"、"大幅提升"而没有数字。
- Results 段的声明范围大于 Methods 实际覆盖的范围。
- 没有独立的 Limitations 段，或把 limitations 藏在末尾一句话里。
- 引入术语未定义，或把"近似等于"写成"等于"。

**图类（B）**
- 用 jet / rainbow 映射定量数据。
- 柱状图 y 轴截断未声明。
- 只画 mean 不画原始散点或误差棒。
- 同一个量在不同图里用不同符号/颜色。
- 位图分辨率不足，放大锯齿清晰可见。
- schematic 未在 caption 明示，被误认为数据图。
- 没有 Source Data 文件。

**表类（C）**
- 不确定度类型不写（审稿人最爱抓这个）。
- 有效数字溢出（`fidelity = 0.996234 ± 0.02`）。

**Methods 类（D）**
- 硬件参数"典型值"一笔带过，没有逐 qubit 数据。
- 经典基线的 bond dimension / 采样数没公布，直接给"慢 N 倍"。
- 没有收敛性研究，拿一个 bond dimension 的结果当终值。
- 只跑了一条路径，没有独立方法交叉验证。

**SI / Extended Data 类（E）**
- 没有负对照。
- 失败尝试只字不提，让读者以为一次成功。
- SI 图表不独立编号，与主稿编号混淆。
- Raw data 不给或以"available upon request"糊弄（顶刊已不接受这句话）。

**代码与数据类（F）**
- 只给 GitHub URL 不给 Zenodo DOI。
- 复现脚本跑不通，或依赖本地绝对路径。
- 未披露"N 次选 1"的挑选过程。
- 未实现被反击声明的原始参数作为基准对比。

**参考文献类（G）**
- 编造 DOI / arXiv ID。
- 用 review 代替原始论文引用。
- 漏引近半年的相关工作。
- 引了已撤稿论文没发现。

**论证逻辑类（H）**
- 把"经典方法跑不动"说成"经典方法不可能跑动"。
- 把部分打破（🟡）包装成完全打破（🔴）。
- 把 asymptotic speedup 写成 wall-clock speedup。
- 循环论证：用未被打破当作不可打破的证据。

**披露类（I）**
- 未披露 LLM 辅助生成的段落。
- CRediT 漏人或把同一人列满所有类目。
- 没有数据/代码可用性声明。

**流程类（通用，不归入 A–J 任一节）**
- 为赶进度绕过强制交叉评审（至少两位独立审查，见前节）。
- 审查意见不落盘、不推送，只在 eacn3 口头说"已改"。
- 编造任何数据或基准结果（引用编造见 G 节）。
- 未复现的结果写成已复现，或避而不谈复现失败。
- 在 Methods / SI 留黑箱、藏参数、掩盖调参过程。
- 用 LLM 生成的"看起来像论文"的段落直接贴进主稿——**每句话必须有实质依据**。

**这是唯一的机会。** 按以上 A–J 全部打钩交货，或者不交。
